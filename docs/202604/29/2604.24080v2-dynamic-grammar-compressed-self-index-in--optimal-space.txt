Abstract 

A compressed self-index stores a string in compressed form while supporting locate queries without de-compression. For highly repetitive strings—arising in web crawls, versioned documents, and genomic collections—static self-indexes can match the δ-optimal lower bound of Ω( δ log (n log σ/ (δ log n)) log n)

bits up to constant factors, where n is the string length, σ is the alphabet size, and δ is the sub-string complexity. Their dynamic counterparts, however, remain scarce: every existing dynamic self-index either fails to attain δ-optimal space, pays at least Θ( log n) time per reported occurrence during locate, or exposes the longest common prefix (LCP) of the text inside its update time. We present the dynamic RR-index , a dynamic grammar-compressed self-index built on the restricted recompression run-length straight-line program (RLSLP). To our knowledge, it is the first dynamic self-index to attain δ-optimal space. The index occupies expected O(δ log (n log σ/ (δ log n)) log n)

bits, answers locate queries in expected O(m + log m log 2 n + occ (log n/ log log n)) time—where m

is the pattern length and occ is the number of occurrences—and supports insertions and deletions of a length-m′ substring in expected amortized O(m′ log 2 n + log 3 n) time, with no dependence on the LCP. On eleven highly repetitive corpora, including a 37 GB Wikipedia dump and a 59 GB human-chromosome collection, the dynamic RR-index is up to 77 × faster than the dynamic r-index on updates and up to 11 × faster than other dynamic indexes on locate.

## 1 Introduction 

Datasets composed of highly repetitive strings—those with many repeated substrings—have grown rapidly in recent years. Typical examples include web pages collected by crawlers [ 14 ], version-controlled documents such as Wikipedia with its complete edit history [ 46 ], and, perhaps most notably, biological sequences such as collections of human genomes [ 42 , 45 ]. These datasets are not static: edits, deletions, and new releases arrive continuously. Yet the methods practitioners rely on most heavily, such as the r-index [ 15 ], a compressed index supporting locate queries, are static : reflecting even a single edit requires rebuilding the index from scratch. There is therefore a strong and growing need to develop dynamic compressed self-indexes for highly repetitive strings, which can be updated efficiently without rebuilding. A practical solution must simultaneously (i) use space proportional to the repetitiveness of the string, (ii) answer locate queries quickly, and (iii) support string insertions and deletions without rebuilding. The third requirement has proven difficult: while compressed 

self-indexes —data structures that represent the string in compressed form while answering locate queries directly—are well understood in the static case, their dynamic counterparts remain challenging to develop. For highly repetitive strings, compressed self-indexes aim to approach the δ-optimal space of O(δ log (n log σ/ (δ log n)) log n) bits established by Kociumaka et al. [ 27 ], where n

A line of specialized self-indexes—grammar-based [ 8, 27 , 26 ], LZ-based [ 7], block-tree–based [ 2, 29 , 27 ], and run-length BWT (RLBWT)–based such as the r-index [ 15 ] and OptBWTR [ 36 , 3 ]—has closed much of this gap, with several attaining δ-optimal space up to constant factors. None, however, supports updates. Only a handful of dynamic self-indexes have been proposed, and each sacrifices at least one of the three requirements (Table 1 in Appendix A gives the full comparison). The dynamic FM-index [ 44 , 31 ] extends the Burrows–Wheeler Transform (BWT) [ 5 ] but uses 

O(n log σ) bits, oblivious to repetitiveness, with an O(log 2+ ϵ n) per-occurrence locate factor. Grammar-based dynamic indexes—the SE-index [ 35 ], TST-index-d [ 40 ], and Gawrychowski et al. [ 16 ]—either pay (log n log ∗ n)2-type factors in locate/update or require Ω( n log n) bits. A recent dynamic r-index [ 38 , 39 ] uses O(r log n) bits ( r the number of BWT runs), but its update time O(( m′ + Lmax ) log n) depends on the maximum value Lmax ≤ n of the longest common prefix (LCP) array—the length of the longest repeated substring—which can grow independently of δ and r as versioned copies accumulate. In short, every existing dynamic self-index either fails to compress to δ-optimal space, pays at least Θ( log n) time per reported occurrence during locate, or exposes the LCP inside update time. We address this gap. 

Our contributions. We present the dynamic RR-index , a dynamic grammar-compressed self-index built on the restricted recompression RLSLP of Jeż [ 19 ] and Kociumaka et al. [ 28 , 27 ]. Our starting observation is that the derivation tree of a restricted recompression admits a com-pact DAG representation whose explicit-node count is tied to δ rather than to r or g. Moreover, insertions and deletions perturb this DAG only along two short popped sequences [ 35 , 18 ], inde-pendently of the maximum LCP value. Combined with two-dimensional range reporting [ 4, 33 ]and with a path-cached ancestor field that accelerates locate (Section 5.6), this design simul-taneously achieves δ-optimal space and a sub-logarithmic per-occurrence locate bound. To our knowledge, the dynamic RR-index is the first dynamic self-index to attain δ-optimal space. Concretely, it occupies expected δ-optimal space— O(δ log (n log σ/ (δ log n)) log n)

bits—and answers locate queries in expected O(m + log m log 2 n + occ ( log n/ log log n)) time, where m is the pattern length and occ is the number of occurrences. Insertions and deletions of a length-m′ substring are supported in expected amortized O(m′ log 2 n + log 3 n) time. We implement and evaluate the dynamic RR-index on eleven highly repetitive corpora, including the 37 GB enwiki Wikipedia dump and the 59 GB chr19 genomic collection, and we show that it is up to 77 × faster than the dynamic r-index on updates and up to 11 × faster than other dynamic indexes on locate, while using working memory consistent with scaling by δ. A C++ reference implementation will be released as open-source together with the camera-ready version.

## 2 Preliminaries 

Basic notation. Let T be a string of length n ≥ 2 over a totally ordered alphabet Σ of size σ = nO(1) . Let |T | = n. We denote the empty string by ε, the i-th character of T by 

T [i], the substring of T from position i to position j by T [i..j ], and the reverse of T by 

reverse (T ) = T [n], T [n − 1] , . . . , T [1] . A prefix (resp. suffix) of T is a substring beginning at position 1 (resp. ending at position n); for a string P ∈ Σ+, T · P denotes concatenation. We use the standard lexicographic order on Σ∗. The set of occurrence positions of P in T

is Occ (T, P ) = {i ∈ { 1, . . . , n − | P | + 1 } | T [i..i + |P | − 1] = P }. For a nonempty set S of integers, min S and max S denote its minimum and maximum. For a path P in a graph, let 

|P| denote the number of edges on P.

Model of computation. We will use base-2 logarithms throughout this paper unless otherwise indicated. Our computation model is a unit-cost word RAM [ 17 ] with multiplication, randomization, and machine word size B = Θ( log n) bits. We measure space in machine words; multiplying by B gives the space in bits. We analyze our randomized algorithms against an oblivious adversary [ 6]: the sequence of updates to T is chosen in advance, without access to the random bits used by our algorithms. 

Substring complexity. Substring complexity [ 43 , 8 ] measures the repetitiveness of a string. For each d ≥ 1, let Sd = {T [i..i + d − 1] | 1 ≤ i ≤ n − d + 1 } be the set of length-d substrings of T . The substring complexity of T is δ = max 1≤d≤n |S d|/d .The δ-optimal space is Θ( δ log n log σδ log n log n) bits: every string can be encoded in the bits, and this bound is tight in n, σ, and δ [27]. 

Grammar-based compression. A grammar-based compression of T is a context-free gram-mar G = ( V, Σ, D, E ) that generates exactly T , where V = {X1, X 2, . . . , X g } is a set of 

nonterminals , Σ is the alphabet of T , and D is a set of production rules Xi → expr i with 

expr i ∈ (V ∪ Σ) ∗. A rule Xi → expr i expands Xi to the sequence expr i. The start symbol 

E ∈ V does not appear on any right-hand side, while every other nonterminal does appear on at least one. To avoid cycles, we require j < i whenever Xj appears in expr i. In addition, 

expr i̸ = expr j for two distinct integers i, j ∈ { 1, 2, . . . , g }.Each nonterminal X ∈ V derives a substring of T ; in particular, the start symbol E

derives T itself. The tree whose root is labeled E and whose leaves spell out T is the 

derivation tree of G. Let val (X) denote the substring of T derived by X ∈ V , and let 

val (X1, X 2, . . . , X d) = val (X1) · val (X2) · . . . · val (Xd) for nonterminals X1, X 2, . . . , X d ∈ V .A straight-line program (SLP) [ 20 ] is a grammar-based compression in which each nonter-minal produces either a character or a pair of nonterminals. A run-length SLP (RLSLP) [34 ]is an SLP whose nonterminals can additionally produce a repetition of a single nonterminal. Specifically, D ⊆ V × (Σ ∪ V +), and each production rule in D takes one of the following forms: (i) Xi → c for c ∈ Σ; (ii) Xi → Xj Xk (j, k < i ) for distinct nonterminals Xj , X k ∈ V ;(iii) Xi → Xj (j < i ) for Xj ∈ V ; (iv) Xi → (Xj )d (j < i ), where (Xj )d denotes d ≥ 2

successive copies of Xj ∈ V . Several algorithms build an RLSLP for a given string, e.g.,

## 3 Restricted Recompression 

Restricted recompression [ 27 , 28 ] is a randomized algorithm that constructs an RLSLP 

GR whose derivation tree is height-balanced (i.e., all children of every node have the same height). Let H denote the height of the derivation tree of GR. For each h ∈ { 0, 1, . . . , H },the nonterminals labeling the nodes at height h, read left to right, form a sequence Sh. In this paper, GR is called restricted recompression RLSLP .Restricted recompression builds the derivation tree bottom-up, processing S0, S 1, . . . , S H

in order. While processing Sh, each newly introduced nonterminal Xi is assigned a value 

assign (Xi) ∈ {− 1, 0, 1} as follows. Let μ(h) = (8 /7) ⌈h/ 2⌉− 1. If |val (Xi)| ≤ μ(h), then 

assign (Xi) is drawn uniformly at random from {0, 1}; otherwise, assign (Xi) = −1. The assign 

values partition Sh into segments. Each segment is one of the following: (i) a maximal run of the same nonterminal Xi with assign (Xi) ∈ { 0, 1}; (ii) two distinct nonterminals 

Xi and Xj with assign (Xi) = 1 and assign (Xj ) = 0 ; or (iii) a single nonterminal Xi with 

assign (Xi) ∈ {− 1, 0, 1}.Figure 1 shows an example of the derivation tree for T = abaaacbaaa . Since the actual values of μ(h) would make the tree too large to display, we use ⌊μ(0) ⌋ = ⌊μ(1) ⌋ = 1 and

## 4 Dynamic RR-index 

The dynamic RR-index is built on the restricted recompression RLSLP GR = ( V, Σ, D, E )

introduced in the previous section. It consists of two components: (i) a directed acyclic graph (DAG) representing the derivation tree, and (ii) a set of points in two-dimensional space corresponding to explicit nodes in the DAG. The details of these two components are described in the following subsection. 

## 4.1 DAG Representation of Derivation Tree and Path Decomposition 

DAG representation. The derivation tree of GR is represented as a directed acyclic mul-tigraph [ 34 , 35 ] D = ( U, E, src , dst , LU , LE ) obtained by merging nodes labeled with the same nonterminal: LU : U → V is a bijection between nodes and nonterminals, E is the set of directed edges; for e ∈ E , src (e) is its source (corresponding to a parent node in the derivation tree) and dst (e) its destination (a child node), and the outgoing edges of ui are in one-to-one correspondence with those of any node labeled LU (ui) in the derivation tree. The label LE (e) ∈ { 0, 1, . . . , n − 1} is j − i, where i and j are the starting positions of the substrings derived from src (e) and dst (e). Let d+(u) denote the outdegree of u. ## 4.2 Point Representation of Explicit Nodes 

To support efficient locate queries, we map each explicit node ui ∈ U expl to a point pi in two-dimensional space at x-coordinate Li and y-coordinate Ri, defined from the production rule of Xi = LU (ui) as follows: if Xi → Xj Xk then Li = reverse (val (Xj )) and Ri = val (Xk); if 

Xi → c for c ∈ Σ then Li = c and Ri = ε; if Xi → (Xj )d with d ≥ 2 then Li = reverse (val (Xj )) 

and Ri = val (( Xj )d−1). Let P be the resulting point set, and let X and Y be Uexpl sorted lexicographically by Li and Ri, with ties broken by ui. This point representation will later allow us to find the relevant explicit nodes for locate queries by two-dimensional range reporting, i.e., by queries that return all points in a given axis-aligned rectangle. ## 4.3 Dynamic Data Structures for the DAG and Point Set 

The DAG D is indexed by DDAG (Π) and the point set P by Dgrid (P).T. Nishimoto and Y. Tabei 7

Data structure for path decomposition. DDAG (Π) consists of a doubly linked list and two types of hash tables [ 11 ]. Intra-path edges are implicit in the list structure; inter-path edges are stored in per-path hash tables. For each explicit node u ∈ U expl with Xi := LU (u), the list stores one element associated with the endpoint u that records: a pair (u, |Pu|) for recovering 

Pu, the production rule Xi → expr i, the height of u, |val (Xi)|, and assign (LU (uj )) for each node uj on Pu (occupying O(H) bits since |Pu| ≤ H + 1 and each label is 2 bits). For each path Pu ∈ Π, the set Eexpl (Pu) = {e ∈ E | src (e) ∈ U expl , dst (e) lies on Pu} is indexed by a per-path hash table whose entry for e has key src (e) and value dst (e) (only distinct key-value pairs are stored); the keys are also linked in a doubly linked list sorted by the height of the corresponding nodes and indexed by a list indexing data structure [ 9]. A global hash table indexes the M production rules with key expr i (encoded in O(B) bits) and value Xi.

Data structure for point set. We index P ⊆ X × Y by Blelloch’s dynamic data structure 

Dgrid (P) [4] for range reporting and point insertion/deletion, which requires O(1) -time comparison of elements of X and Y. We maintain order maintenance data structures [10 ]on X and Y to provide O(1) -time comparison and O( log M )-time insertion/deletion, and list indexing data structures for O(log M )-time access; each coordinate is a pointer to the corresponding list element of Π, so each point uses O(B) bits. Each list element of DDAG (Π) 

in turn stores pointers into the list indexing and order maintenance structures for X and Y.Range reporting takes O(log M + rocc (log M/ log log M )) time where rocc is the number of reported points, and a point insertion or deletion takes amortized O(log M ) time. 

## 4.4 Space Complexity 

The doubly linked list contributes O(M (H + B)) bits; the global hash table, the per-path hash tables (which hold O(M ) distinct key-value pairs in total since every DAG node has at most two children) together with their list indexing structures, and Dgrid (P) with its auxiliary structures each contribute O(M B ) bits. Summing, the dynamic RR-index occupies 

O(M (H + B)) bits. Since B = Θ( log n) and E[M ] = O(δ log n log σδ log n ), the total space is 

δ-optimal in expectation under the assumption that H = O(log n).

## 5 Locate Query Algorithm 

This section presents the algorithm answering a locate query for a pattern P of length m,establishing the baseline complexity bound (Theorem 5); Section 5.6 then improves the per-occurrence factor to log n/ log log n via ancestor-path caching (Theorem 6). For m ≥ 2,the algorithm proceeds in four steps, each detailed in a subsection below: from the derivation tree we extract a specific core [32 ] of P as the popped sequence [12 , 18 ] of ρ runs, build ρ

axis-aligned rectangles in P, apply range reporting to obtain the primary occurrences [ 35 ], and convert these into Occ (T, P ) by traversing the DAG to the root. The case m = 1 is handled separately in Section 5.5.

## 5.2 Step (ii): Computing the Rectangles from the Popped Sequence 

Primary occurrences and their rectangles. A primary occurrence [ 35 ] of P for a split position ℓ ∈ { 1, 2, . . . , m − 1} is a pair (ui, ℓ ) with ui ∈ U expl such that (i) reverse (P [1 ..ℓ ]) is a prefix of Li and (ii) P [ℓ + 1 ..m ] is a prefix of Ri; that is, ui is an explicit node where an occurrence of P crosses the first boundary between substrings derived from ui’s production rule. We write pOcc (P, ℓ ) ⊆ U expl for the set of such ui at split position ℓ. Recall from Section 4.2 that ui is mapped to the point pi with x-coordinate Li and y-coordinate Ri.The two prefix conditions constrain Li and Ri, respectively, so pOcc (P, ℓ ) coincides with the set of points of P in the axis-aligned rectangle rect ℓ = [ x, x ′] × [y, y ′], where x, x ′ ∈ X are the smallest and largest elements whose x-coordinate has reverse (P [1 ..ℓ ]) as a prefix, and 

y, y ′ ∈ Y are the analogous bounds for P [ℓ + 1 ..m ] (if any of the four does not exist, rect ℓ is empty).

## 6 Update Operations 6.1 Insertion Operation 

Given a string P ′ ∈ Σ+ of length m′ = O(n) and a position s ∈ { 1, 2, . . . , n }, the insertion operation produces T ′ by inserting P ′ into T at position s and updates the dynamic RR-index so that its RLSLP derives T ′. For simplicity, we focus on 2 ≤ s ≤ n − 1, so that T ′ is the concatenation of T [1 ..s − 1] , P ′, and T [s..n ]; the boundary cases s = 1 and s = n

are analogous. Following the standard strategy of dynamic grammar-compressed data structures [ 40 , 35 , 37 , 34 ], the update proceeds in three phases (see Appendix D for details). 

Phase 1: constructing the derivation tree of GR′

. The first phase constructs the derivation tree of an RLSLP GR′

= ( V′, Σ′, D′, E ′) deriving T ′ by restricted recompression, computing 

(H′ +1) sequences S′0, S ′1, . . . , S ′H′

bottom-up; production rules are shared with GR whenever the right-hand sides agree. For each nonterminal in GR′

\ G R, we insert the corresponding node into the DAG and update the dynamic data structures. A naive construction is 

Ω( n + m′) since the tree has n + m′ leaves. We reduce this cost using the popped sequences 

QL and QR of T [1 ..s − 1] and T [s..n ]: since each popped sequence is a core of its source, 

QL and QR are embedded in the derivation tree of GR′

at positions 1 and s + m′ as subtrees that need not be reconstructed. Concretely, (i) compute QL and QR by [ 12 ] in 

O(H) time; (ii) build S′0, . . . , S ′H′

in O(|T diff |) time, where Tdiff collects (A) the ancestors of the embedded subtree roots, (B) the m′ leaves on T ′[s..s + m′ − 1] , and (C) their ancestors. Figure 5 illustrates the embeddings. The phase runs in expected amortized 

O(|T diff | + |I expl | max {H, H ′, log M } log M + |I impl |) time, where Iexpl (resp. Iimpl ) is the set of explicit (resp. implicit) nodes inserted into the DAG; see Appendix D.1. 

Phase 2: removing obsolete nonterminals. For each nonterminal in GR \ G R′

, remove the corresponding node from the DAG and update the dynamic data structures. This phase takes expected amortized O(|R expl | log M + |R impl |) time; see Appendix D.2 for details. Here, Rexpl 

(respectively, Rimpl ) is the set of explicit (respectively, implicit) nodes whose nonterminals do not appear in the derivation tree of GR′

.

Phase 3: updating ancestor-path caches and coordinate prefixes. For each explicit node in Iexpl ∪ U change , update the additional information stored in the corresponding element of the doubly linked list for Π. ## B.2 Construction Algorithm 

Restricted recompression proceeds as follows. At level 0, for each position s ∈ [1 , n ], the algorithm introduces a fresh nonterminal Xis with rule Xis → T [s] and determines assign (Xis ).The sequence S0 is then Xi1 , X i2 , . . . , X in . Here, for two characters T [s] and T [s′], Xis = Xis′

if and only if T [s] = T [s′].For each h ≥ 1, Sh is built from Sh−1 in two steps. In the first step, Sh−1 is decomposed into segments F1, F 2, . . . left-to-right as follows. Suppose Sh−1[1 ..s − 1] has been decomposed into F1, . . . , F j−1; then the next segment Fj , starting at position s, is: 

1. Fj = Sh−1[s..s + 1] if h is even, assign (Sh−1[s]) = 1 , and assign (Sh−1[s + 1]) = 0 ;

2. Fj = Sh−1[s..s + ℓ − 1] , the longest run of Sh−1[s] starting at position s, if h is odd and 

assign (Sh−1[s]) ̸ = −1;

3. Fj = Sh−1[s] otherwise. In the second step, for each segment Fj of Sh−1, the algorithm introduces a fresh nonterminal Xij with rule Xij → Fj and determines assign (Xij ). Concatenating these nonterminals in order yields Sh. Here, for two segments Fj and Fj′ , Xij = Xij′ if and only if Fj = Fj′ .These two steps are applied for h = 1 , 2, . . . until the sequence has length one; the final sequence is SH , with H ≥ 1 since |S0| = n ≥ 2.The algorithm outputs an RLSLP GR = ( V, Σ, D, E ), where V collects the nonterminals introduced across S0, S 1, . . . , S H ; Σ is the alphabet of T ; D is the set of production rules created above; and the start symbol E is the unique nonterminal in SH .By Corollary V.11 of [ 27 ], the algorithm terminates with probability 1, and its expected running time is O(n).

## C.5 Algorithm for Constructing the Popped Sequence 

To explain how to construct the popped sequence, we introduce a rule query : given a right-hand side expr , return the nonterminal X ∈ V such that X → expr ∈ D , or ⊥ if no such nonterminal exists. This query is based on the following observation. If X corresponds to an explicit node, then it is stored in the global hash table as the value associated with key expr . Otherwise, X corresponds to an implicit node u ∈ U ; in this case, expr is a single nonterminal X′ ∈ V , and the node corresponding to X′ is the child of u on a path in Π.Using this observation, a rule query can be answered in O(1) time using the data structures maintained by the dynamic RR-index. The popped sequence of P is built bottom up using rule queries, as in restricted recom-pression. If the construction would require creating a new nonterminal, the algorithm aborts. In this case, Occ (T, P ) = ∅ follows from the definition of the core. Otherwise it runs in 

O(∑ ˆHh=0 |ShP |) time. Kociumaka showed E[∑ ˆHh=0 |ShP |] = O(m) (Lemma V.10 in [ 27 ]), so the expected running time is O(m).

## D Details of Insertion Operation D.1 Details of Phase 1 

Update of dynamic RR-index. Every time a new production rule Xi → expr i is established, a new node ui of height h corresponding to Xi is inserted into the DAG. If ui is explicit, then a new path P = ui is inserted into Π, and a new directed edge e from ui to the node uj

corresponding to each nonterminal Xj in expr i is inserted into Eexpl (Pj ), where Pj ∈ Π is the path containing uj .Otherwise (i.e., ui is implicit), expr i consists of a single nonterminal Xj′ , the corresponding node uj′ is the first node of a path P′ ∈ Π, and the path is extended by connecting ui and P′.To reflect the changes of Π and Eexpl (Pj ), we need to update the following data structures: 

The global hash table. If ui is explicit, Xi and the binary representation of expr i are inserted into the hash table as a value and its key, respectively. This insertion takes expected amortized O(1) time [11]. 

The list element corresponding to P. If ui is explicit, then create a new element corres-ponding to P, append it to the doubly linked list for Π, and associate an empty hash table with it. This element stores the following five pieces of information: (i) a pair (ui, 0) 

for recovering P from the element, (ii) expr i, (iii) h, (iv) |val (Xi)|, and (v) assign (Xi).Here, |val (Xi)| is computed by summing up the length of the substring derived from each nonterminal Xj in expr i if expr i is a sequence of nonterminals. Otherwise, |val (Xi)| = 1 .

|val (Xi)| can be computed in O(1) time using Lemma 4. Therefore, the list element can be updated in O(1) time. 

The list element corresponding to P′. If ui is implicit, then the second component of the pair stored in the list element is increased by one, and assign (Xi) is appended to the list element. These updates take O(max {1, h/B }) time. 

The per-path hash table corresponding to Pj . The key and value of the directed edge e

are inserted into the hash table if the hash table does not have the key. This insertion takes expected amortized O(1) time. In addition, the key of e is inserted into the doubly linked list for the keys at an appropriate position. The list indexing data structure can support a random access to the keys and an insertion into the list in O(log M ) time [ 9]. To compute the appropriate insertion position, we need O( log M ) random accesses to the keys. Therefore, the update of the doubly linked list takes O(log 2 M ) time in total. 

The dynamic data structures for X . If ui is explicit, then Li is inserted after the largest element in X preceding the x-coordinate Li of ui, and the list indexing and order maintenance data structures for X are updated accordingly in O( log M ) time.
